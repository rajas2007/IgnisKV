# ADR-004: Request Pipeline

## Status
Accepted

## Date
2026-07-02

## Context
In database engineering, request execution lifecycle code is easily coupled with networking, serialization, or storage formats. If a network socket handler directly reads bytes, parses commands, executes storage updates, and sends back error strings in a single code block, the code becomes highly fragile and difficult to extend, test, or modify.

## Problem Statement
How should request processing be structured to guarantee separation of concerns, testability, and flexibility for future networking protocols and storage adapters?

## Decision
We enforce a strict, unidirectional processing pipeline for all command requests:

`Client -> Input -> Parser -> Dispatcher -> Command Handler -> Store -> Response`

Each stage in this pipeline has exactly one responsibility:
1. **Input (Reader)**: Reads raw bytes or lines from a transport medium (CLI stdin in `v0.1`, TCP sockets in `v0.2`).
2. **Parser**: Syntactically tokenizes raw input into key arguments and returns a structured Command object.
3. **Dispatcher**: Resolves the command string to a registry-registered Handler and invokes it.
4. **Command Handler**: Validates arguments, coordinates transactions, and queries/modifies the database store.
5. **Store**: Processes thread-safe memory reads/writes.
6. **Response (Writer)**: Returns structured output types that are serialized back to the output medium.

## Rationale
This pipeline isolates network transport details (such as TCP/RESP) from core database engine logic. The parser can be verified independently without spinning up a socket listener. The database engine remains unaware of whether commands are generated by a local console shell session, a TCP port, or a test runner.

## Alternatives Considered
* **Alternative A: Monolithic request loop**: Read data directly in a loop, extract key-value operations with string matching, and mutate a global map in the same routine. Rejected because it is impossible to unit test components individually, lacks safety, and hinders adding multi-client support.
* **Alternative B: Bi-directional event-bus routing**: Use asynchronous Go channels to route messages back and forth between network layers and storage layers. Rejected because it introduces unnecessary concurrency complexity, deadlock paths, and allocation overhead for basic operations.

## Consequences
* **Positive**: High test isolation, simple component swapping (e.g., swapping console input for TCP input in v0.2), and clear code paths.
* **Negative**: Introduces minor allocation overhead to pass Command and Response structs along the pipeline.

## Future Evolution
This pipeline layout remains consistent throughout the project. When we introduce multi-client concurrency in `v0.4`, we will run the pipeline inside worker goroutines, using the same parser, dispatcher, and store interfaces.
